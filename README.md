# rag-lab

A lab for building and **measuring** RAG on models you run yourself. Point it at your markdown
docs (a folder or a GitHub repo), ask questions in plain language, and get cited answers —
generated by a local model via [Ollama](https://ollama.com), no API keys required to run.

But answering is table stakes. Every component (loader, chunker, embedder, retriever, LLM,
agent) is swappable behind a clean interface, and a committed eval harness lets you **prove**
a change helped instead of guessing. Swap a knob, re-run the golden set, watch the metric move.
Agentic retrieval is a first-class, glass-box citizen: watch a small model orchestrate its own
retrieval, and measure whether that actually beats a plain pipeline.

## Requirements

- Python 3.11+ and [uv](https://docs.astral.sh/uv/)
- A running [Ollama](https://ollama.com) with the default models pulled:

```bash
ollama pull nomic-embed-text   # embeddings
ollama pull llama3.2:3b        # answer generation
```

## Install

```bash
git clone https://github.com/aboydnw/local-llm-rag.git
cd local-llm-rag
uv sync
```

Commands below use `uv run rag-lab`; drop the `uv run` prefix if you install it on your PATH.

## Quickstart

```bash
uv run rag-lab config init                      # write a default rag.yml
uv run rag-lab ingest ./path/to/markdown        # build a local index (rag.db)
# ...or point straight at a GitHub repo:
uv run rag-lab ingest --repo developmentseed/titiler

uv run rag-lab ask "how do I add a custom raster source?"
```

`ask` retrieves the most relevant chunks (hybrid keyword + vector search), hands them to the
local model, and prints an answer followed by a numbered source list. Use `--show-chunks` to
see what was retrieved, and `rag-lab inspect chunks` to debug the index.

Configuration lives in `rag.yml` — swap the chunker, embedder, retriever weights, or LLM there.
This is your local working config (gitignored once created); `config init` writes a default to
start from. The ready-made setups under `examples/` are versioned and meant to be copied or
pointed at, not edited in place.

## Agent mode — let the model drive retrieval

By default, every question goes through one fixed retrieval pass. Agent mode flips that: the
local model runs a ReAct loop (Thought → Action → Observation), choosing retrieval tools step
by step — semantic search, keyword search, listing the corpus, or reading a whole document —
until it decides it has enough evidence to answer.

```bash
uv run rag-lab ask "how do I add a custom raster source?" --agent --show-trace
uv run rag-lab eval --golden golden.yml --report report.md --agent
```

`--show-trace` prints the reasoning loop so you can watch the model work. In eval mode the
standard metrics score the chunks that reached the final answer, so agent runs land on the
same leaderboard as fixed-retriever runs — you can measure whether orchestration actually
beats plain hybrid retrieval on your corpus, and what it costs (`tool_calls` and `llm_calls`
are tracked per run).

Agent mode is slower — several LLM calls per question instead of one — and small models
sometimes wander. That's the point: it's a glass-box way to learn how agents behave, with an
eval harness to keep the conclusions honest. Set `agent.enabled: true` in `rag.yml` to make
it the default, and tune `max_steps`, `final_k`, the tool list, or the agent's own prompt
(`agent.instructions`) to see how its behavior changes.

## Evaluating quality

rag-lab ships with an eval harness so you can measure changes rather than guess at them.

```bash
rag-lab eval --golden golden.yml --report report.md
rag-lab eval --golden golden.yml --repeat 5          # mean ± stdev over 5 passes
rag-lab eval --set-baseline 20260721-153000          # pin a run as the baseline
rag-lab eval --golden golden.yml                     # auto-diffs vs the pinned baseline
```

The harness scores retrieval (recall@k, MRR) and answer quality (keyword coverage, plus an
optional LLM-graded judge via DeepEval — `eval.deepeval: true` in `rag.yml`, judged through
Ollama like everything else). No API keys required.

Every eval writes into a persistent **run store** (`.rag-lab/runs/`), shared by the CLI,
Studio, and the MCP server. Each run records its scores plus provenance: config hash,
rag-lab git sha, and golden-set hash. Pin one run as the **baseline** and every later run
prints a delta table against it; configure `eval.gates` in `rag.yml` (max allowed drop per
metric) and the eval exits non-zero on regression — that exit code *is* the CI gate (see
`examples/github-actions-eval-gate.yml`). Local models are noisy, so `--repeat N` reruns
the set and reports mean ± stdev; gates compare means, which keeps you from chasing
run-to-run phantoms.

`examples/devseed-oss/` holds a golden set covering titiler, eoAPI, and stac-fastapi.
`eval-reports/` is the project's eval trail — every report is committed so you can see how a
change moved the metrics.

### Eval tools over MCP

`rag-lab mcp` (optional extra: `uv sync --extra mcp`) serves the harness as MCP tools —
`run_eval`, `list_runs`, `compare_runs`, `get_failures`, `add_golden_case` — over stdio.
Point any MCP client (Claude Code, Cline, …) at it and the agent gets a self-verifying
loop: change a knob in `rag.yml` (or pass `config_overrides`), call `run_eval`, read the
delta vs the pinned baseline, and keep or revert the change based on measured evidence.
`add_golden_case` lets the agent grow the golden set from questions the corpus answered
badly in real use.

## Studio — the visual cockpit

`rag-lab studio` is a local web interface for turning every knob, editing the eval set, and
seeing results — built to help you understand and fine-tune the system by experiment.

```bash
uv sync --extra studio        # one-time: install the studio dependencies
uv run rag-lab studio         # launches a browser at http://localhost:8501
```

It runs fully local (Streamlit, no JS toolchain, no API keys) and has four surfaces:

- **Sidebar config knobs** — chunker, embedder, LLM, retriever, and agent settings, each tagged
  🟢 *cheap* (re-run eval only) or 🟠 *needs re-index* (rebuild the index). Save back to `rag.yml`.
  Agent mode gets its own section: toggle it on, cap the steps, pick which tools the model may
  use, and edit the ReAct prompt that teaches it how to reason.
- **Ask playground** — one question, live; see the retrieved chunks, the assembled prompt, and
  the generated answer. With the agent on, you get the full reasoning trace: every Thought,
  tool call, and observation, plus what it cost.
- **Evaluate** — run the golden set through the current config and see scored results.
- **Runs** — every eval run is saved with its config; a leaderboard lets you compare any two
  and see which knob moved which metric. Agent traces are saved per question, so when an item
  scores zero you can open the trace and see what the agent actually did.
- **Golden set editor** — add/edit golden questions in a form; writes back to `golden.yml`.

Studio caches each index keyed by its corpus + chunker + embedder under a gitignored
`.rag-lab/` directory, so repeating a config reuses its index instead of rebuilding. The
studio dependencies are optional — the core CLI installs and runs without them.
